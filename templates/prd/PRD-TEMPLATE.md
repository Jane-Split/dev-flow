# dev-flow 需求文档（PRD）模板

> **使用说明**：本模板专为 dev-flow AI 开发工作流设计。产品经理按此模板输出需求文档，开发者使用 `/dev-flow -subagent` 即可准确无误地完成需求。
>
> **填写规则**：
> - `【必填】`：必须填写，否则 dev-flow 会提问澄清
> - `【选填】`：建议填写，有助于提高开发准确性
> - `【自动】`：由 dev-flow 自动生成，产品经理无需填写
> - 不要删除任何章节标题，即使没有内容也填写"无"

---

## 一、需求概述

### 1.1 基本信息 【必填】

| 项目 | 内容 |
|------|------|
| **需求标题** | 用一句话描述需求，不超过 30 字 |
| **需求类型** | 新功能 / 功能增强 / Bug 修复 / 重构 / 性能优化 |
| **优先级** | P0(紧急) / P1(高) / P2(中) / P3(低) |
| **需求背景** | 为什么要做这个需求？解决什么业务问题？ |
| **目标用户** | 谁会使用这个功能？（角色名称） |
| **期望上线时间** | YYYY-MM-DD |

### 1.2 业务价值 【选填】

| 项目 | 内容 |
|------|------|
| **业务痛点** | 当前业务中存在什么问题？ |
| **预期收益** | 完成后能带来什么业务价值？ |
| **成功指标** | 如何衡量需求完成？（如：效率提升 X%、错误率降低 X%） |

## 二、功能描述

### 2.1 核心功能点 【必填】

> 每个功能点用一句话描述，dev-flow 会基于此进行任务拆分。

| # | 功能点 | 描述 | 优先级 | 复杂度 |
|---|--------|------|--------|--------|
| 1 | 功能名称 | 具体做什么，输入什么，输出什么 | P0/P1/P2 | 高/中/低 |
| 2 | ... | ... | ... | ... |

**填写示例**：
| # | 功能点 | 描述 | 优先级 | 复杂度 |
|---|--------|------|--------|--------|
| 1 | 不合格品登记 | 记录不合格品信息，关联产品、批次、检测记录 | P0 | 中 |
| 2 | 审批流程 | 提交审批，支持多级审批和驳回 | P0 | 高 |
| 3 | 处置跟踪 | 记录处置方式（返工/报废/让步接收），跟踪处理进度 | P1 | 中 |
| 4 | 统计报表 | 按产品/时间/原因统计不合格品数据 | P2 | 中 |

### 2.2 业务流程 【必填】

> 描述用户操作的核心业务流程。dev-flow 会基于此设计 Service 层逻辑。

```
用户操作流程：
1. 用户在【页面/接口】执行【操作】
2. 系统校验【校验规则】
3. 系统执行【业务逻辑】
4. 系统返回【结果】
```

**填写示例**：
```
不合格品登记流程：
1. 质检员在"不合格品登记"页面填写不合格品信息（产品、批次、缺陷描述）
2. 系统校验：产品是否存在、批次是否有效、必填项是否完整
3. 系统保存不合格品记录，状态为"待审批"
4. 系统自动提交审批流程（调用 workflow 服务）
5. 系统返回登记成功，显示不合格品编号
```

#### 功能点 1：{功能名称}

| 字段名 | 类型 | 必填 | 说明 | 校验规则 | 示例值 |

| id | Long | 自动 | 主键 | 自动生成 | 1 |
| recordCode | String | 是 | 不合格品编号 | 自动生成，格式：NC-{日期}-{序号} | NC-20260526-001 |
| productId | Long | 是 | 关联产品ID | 必须存在 | 1001 |
| batchNo | String | 是 | 批次号 | 不超过 50 字符 | BATCH-20260526 |
| defectType | String | 是 | 缺陷类型 | 枚举值 | 外观缺陷 |
| description | String | 是 | 缺陷描述 | 不超过 500 字符 | 表面有划痕 |
| status | Integer | 自动 | 状态 | 枚举：0待审批/1审批中/2已通过/3已驳回 | 0 |

### 2.4 接口需求 【选填】

> 如果有明确的 API 设计要求，在此描述。dev-flow 会基于此设计 Controller。

| # | 接口名称 | HTTP 方法 | 路径 | 说明 |
|---|----------|-----------|------|------|
| 1 | 创建不合格品 | POST | /api/non-conforming-products | 提交不合格品登记 |
| 2 | 查询不合格品详情 | GET | /api/non-conforming-products/{id} | 根据ID查询 |
| 3 | 分页查询 | GET | /api/non-conforming-products/page | 支持条件筛选 |

### 2.5 状态流转 【选填】

> 如果功能涉及状态变更，描述状态流转规则。dev-flow 会基于此设计 Enum 和状态机。

```
状态流转图：
[待审批] --提交审批--> [审批中]
[审批中] --审批通过--> [已通过]
[审批中] --审批驳回--> [已驳回]
[已驳回] --重新提交--> [审批中]
[已通过] --处置完成--> [已关闭]
```

| 当前状态 | 操作 | 目标状态 | 触发条件 |
|----------|------|----------|----------|
| 待审批 | 提交审批 | 审批中 | 用户点击提交 |
| 审批中 | 审批通过 | 已通过 | 审批人确认 |
| 审批中 | 审批驳回 | 已驳回 | 审批人驳回，需填写驳回原因 |
| 已驳回 | 重新提交 | 审批中 | 用户修改后重新提交 |

## 三、约束条件

### 3.1 技术约束 【必填】

| 约束类型 | 说明 |
|----------|------|
| **涉及服务** | 这个需求会影响哪些微服务？（如：quality-management、workflow、basedata） |
| **跨服务调用** | 是否需要调用其他服务的接口？（如：需要查询产品信息、启动审批流程） |
| **公共模块变更** | 是否需要在 common-bean 中新增类？（如：新增共享 DTO、Enum） |
| **数据库变更** | 是否需要新建表或修改现有表？ |
| **事务要求** | 是否需要分布式事务？（如：创建不合格品 + 启动审批流程需要保证一致性） |
| **并发要求** | 是否有并发场景？（如：同一批次不允许重复登记） |
| **性能要求** | 是否有性能指标？（如：列表查询 < 500ms） |
| **权限要求** | 哪些角色可以操作？（如：质检员可登记，质量主管可审批） |

### 3.2 业务约束 【选填】

| 约束类型 | 说明 |
|----------|------|
| **数据规则** | 如：不合格品编号每天从 001 开始递增 |
| **审批规则** | 如：审批超过 3 天未处理自动提醒 |
| **业务规则** | 如：已关闭的不合格品不允许修改 |

### 3.3 兼容性约束 【选填】

| 约束类型 | 说明 |
|----------|------|
| **接口兼容** | 是否需要兼容旧版接口？ |
| **数据迁移** | 是否需要迁移旧数据？ |
| **前端兼容** | 是否需要兼容旧版页面？ |
